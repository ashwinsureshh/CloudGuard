What Problem Are We Solving?
The motivation behind CloudGuard

Cloud applications run on networks. Every API call, login, and file transfer is a network flow — a packet of data travelling between machines. Most traffic is normal, but some is malicious. A traditional firewall blocks known bad IPs, but modern attacks come from new sources, use legitimate-looking patterns, and evolve constantly.

Types of attacks CloudGuard detects

AttackWhat the attacker doesWhat it looks like in traffic
DDoSFloods the server with millions of fake requests to crash itHuge spike in incoming packets, tiny responses
PortScanProbes your machine to find open "doors" (services)Many very short connections to different ports
BruteForceTries thousands of passwords per second (SSH, RDP)Many repeated connections to port 22/3389, mostly rejected
InfiltrationAn attacker with partial access tries to move deeperUnusual internal traffic patterns
BENIGNNormal users browsing, calling APIs, downloading filesBalanced packet counts, normal durations
💡
CloudGuard's approach

Train a machine learning model on millions of labelled historical network flows (already marked as attack or benign), then run that model on live traffic and alert the security team in real time — even for attack patterns never explicitly programmed.

Network Intrusion Detection
Core concept — what is a network flow?

A network flow is a statistical summary of a connection between two machines. Instead of logging every byte (terabytes of data), we log a summary of what happened:

example flow record
# A BruteForce SSH attempt looks like this:
src_ip       = "192.168.1.101"  # attacker's IP
dst_port     = 22               # SSH port
duration     = 0.003            # connection lasted 3ms (rejected fast)
fwd_packets  = 450             # attacker sent many packets
bwd_packets  = 2               # server barely replied (wrong password)
flag_count   = 8               # many TCP RST flags (reset connections)

# A human logging in normally looks like this:
dst_port     = 22
duration     = 45.2            # session lasted 45 seconds
fwd_packets  = 12              # typed a few commands
bwd_packets  = 10              # server responded normally

The ML model learns these statistical fingerprints for each attack type from millions of labelled examples in the CIC-IDS2017 dataset.

📦
CIC-IDS2017 Dataset

Created by the Canadian Institute for Cybersecurity. Contains ~2.8 million labelled network flows, 7 attack categories + benign traffic, and 80+ features per flow (packet size stats, timing, TCP flag counts, etc.). The standard academic benchmark for IDS research.

ML Pipeline — preprocess.py
Cleaning and preparing raw data for training

Step 1 — Drop bad rows

ml/preprocess.py
def clean_data(df):
    df = df.drop_duplicates()                   # remove exact duplicate rows
    df = df.replace([np.inf, -np.inf], np.nan)  # infinity → NaN
    df = df.dropna()                            # drop rows with missing values
    return df
⚠️
Why do infinite values appear?

Some features are ratios (e.g., bytes per second). If a connection lasted 0 seconds, that's division by zero → infinity. These rows are mathematically broken and must be removed before training.

Step 2 — Encode text labels to numbers

ml/preprocess.py
LabelEncoder().fit_transform(df["Label"])

# Converts:
"BENIGN"       → 0
"BruteForce"   → 1
"DDoS"         → 2
"Infiltration" → 3
"PortScan"     → 4

ML models only understand numbers — not strings. LabelEncoder assigns an integer to each unique class name.

Step 3 — Normalise feature values (StandardScaler)

ml/preprocess.py
scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)  # compute mean/std FROM training data
X_test_scaled  = scaler.transform(X_test)       # apply SAME mean/std to test data

❌ Without scaling

packet_size ranges 0–65535
flag_count ranges 0–5

The model focuses almost entirely on packet_size because its numbers are bigger — even if flag_count is more informative.

✅ With StandardScaler

Both features rescaled to mean=0, std=1.

Every feature gets an equal chance to influence the model. Training is faster and more accurate.

🚨
Critical rule — never refit on test data

fit_transform computes mean/std and applies it. transform only applies — it does NOT recompute. If you refit the scaler on test data, you're using "future information" during training. This is called data leakage and makes your accuracy numbers fake.

ML Pipeline — train_model.py
Training the Random Forest classifier

Train / Test Split

ml/train_model.py
X_train, X_test, y_train, y_test = train_test_split(
    X, y,
    test_size=0.2,       # 20% held out for evaluation
    random_state=42,    # reproducible split
    stratify=y          # keep class ratios the same in both halves
)

stratify=y is important: if DDoS is 15% of the dataset, it will be exactly 15% in both train and test — not accidentally 5% in one and 25% in the other.

What is a Random Forest?

A Random Forest is an ensemble of Decision Trees.

A single Decision Tree splits data by asking yes/no questions:

Is dst_port == 22? YES ──▶ Is fwd_packets > 300? YES ──▶ BruteForce (high confidence) NO ──▶ BENIGN (moderate confidence) NO ──▶ Is duration < 0.001? YES ──▶ PortScan (likely) NO ──▶ continue splitting...

A Random Forest builds n_estimators=100 such trees, each trained on a random subset of data and features. For a new flow, all 100 trees vote, and the majority wins. This dramatically reduces overfitting.

ml/train_model.py
clf = RandomForestClassifier(
    n_estimators=100,   # 100 decision trees
    max_depth=20,       # each tree max 20 levels deep
    n_jobs=-1,          # use ALL CPU cores in parallel
    random_state=42     # reproducible results
)
clf.fit(X_train, y_train)

💡
Why save the scaler and label encoder too?

At inference time, you MUST scale live data using the exact same mean/std computed during training. If you refit, the scaling will be different and predictions will be garbage. Saving and reloading these objects guarantees consistency.

Apache Kafka
High-speed message streaming between services

Apache Kafka is a distributed message broker — a high-speed pipeline for streaming data between systems. Think of it as a postal system for data:

Producer
writes messages
→
Kafka Topic
message queue
→
Consumer
reads messages

Why not just use a database or REST calls?

MethodProblem at high volume
REST API (one HTTP request per flow)Server gets overwhelmed at 100,000 flows/sec
Database polling (check every second)Slow, adds heavy DB load, not real-time
Kafka✅ Handles millions of events/sec; consumers read at their own pace without overloading the producer

Core Kafka concepts

TermAnalogyIn CloudGuard
TopicA named channel (like a YouTube channel)network-flows — all flow events go here
ProducerSomeone posting videoskafka_producer.py
ConsumerA subscriber watching videosbackend/app.py (not yet implemented)
BrokerYouTube's serversOne Kafka server on port 9092
ZookeeperYouTube's coordination systemManages the Kafka broker metadata
OffsetWhich video you last watchedWhere the consumer left off — survives restarts

Threading — two things at once

backend/app.py
# Python runs one thing at a time normally.
# We need BOTH running simultaneously:
#   1. Flask handling HTTP requests
#   2. simulate_traffic() generating alerts in a loop

thread = threading.Thread(target=simulate_traffic, daemon=True)

# daemon=True: if main program exits, kill this thread too

REST APIs vs WebSockets
Two ways for browser and server to communicate

❌ HTTP Polling (naive approach)

Browser asks "any new alerts?" every second.
Server responds — even if nothing changed.

At 1000 users × 1 req/sec = 1000 wasted requests/sec. Latency = up to 1 full second.

✅ WebSocket (CloudGuard's approach)

One persistent connection per user.
Server pushes data instantly when it's ready.

No wasted requests. Latency = milliseconds. Dashboard feels truly live.

HTTP (request-response — closes after each reply): Browser → "GET /api/alerts" → Server → response → connection closed [1 second later] Browser → "GET /api/alerts" → Server → response → connection closed WebSocket (persistent — server pushes any time): Browser → handshake → connection stays open forever Server → "new_alert" event → Browser updates instantly Server → "stats_update" → Browser updates instantly [connection stays open, no polling needed]

CloudGuard's REST endpoints (for initial load)

GET/api/health{"status": "ok"}
GET/api/statsTotal flows, attacks detected, benign count

Frontend Dashboard
React — component-based real-time UI

React component tree

App.js (owns state: stats, alerts, connected) │ └── Dashboard.jsx (receives data as props) ├── StatCard × 4 (Total Flows / Attacks / Benign / Rate) ├── BarChart (attack type breakdown) ├── PieChart (traffic distribution) └── AlertsTable (live feed of last 15 alerts)

useState — reactive variables

frontend/src/App.js
// Creates a variable AND a setter function
const [stats, setStats] = useState({ total_flows: 0, attacks_detected: 0 })

// WRONG — doesn't trigger re-render:
stats.total_flows = 5

// CORRECT — React sees the change and re-renders the UI:
setStats(newData)

useEffect — connecting to the WebSocket

frontend/src/App.js
useEffect(() => {
    const socket = io(API_URL)   // connect once on mount

    socket.on('connect',      () => setConnected(true))
    socket.on('disconnect',   () => setConnected(false))
    socket.on('stats_update', data  => setStats(data))
    socket.on('new_alert',    alert => {
        setAlerts(prev => [alert, ...prev].slice(0, 50))
        //         ↑ prepend  ↑ spread existing   ↑ keep latest 50
    })

    return () => socket.disconnect()   // cleanup on unmount
}, [])  // [] = run only once when component first loads

Docker & Docker Compose
Running the whole system with one command

A container packages your app + all its dependencies into an isolated unit that runs identically on any machine. No more "it works on my laptop" problems.

TermMeaning
ImageBlueprint/recipe (built from a Dockerfile)
ContainerA running instance of an image (like a process from an executable)
Volume mountLinks a folder on your Mac into the container — files shared both ways
Docker ComposeTool to define and run multiple containers together from one YAML file

Our docker-compose.yml explained

docker-compose.yml
services:
  zookeeper:                      # manages kafka broker metadata
    image: confluentinc/cp-zookeeper:7.4.0
    ports: ["2181:2181"]         # host:container

  kafka:
    depends_on: [zookeeper]      # start zookeeper FIRST
    ports: ["9092:9092"]
    environment:
      KAFKA_ZOOKEEPER_CONNECT: zookeeper:2181
      # "zookeeper" resolves to that container's IP (Docker DNS)

  backend:
    build: ./backend             # build image from ./backend/Dockerfile
    depends_on: [kafka]
    volumes:
      - ./backend/model:/app/model  # share model .pkl files into container

  frontend:
    build: ./frontend
    depends_on: [backend]
    environment:
      REACT_APP_API_URL: http://localhost:5001
🚀
One command to run everything

docker-compose up --build — builds all images, starts all 4 containers in the right order, and wires their networking together automatically.

End-to-End Data Flow
A complete trace from CSV to dashboard pixel
1
Train (runs once, offline)
Download CIC-IDS2017 CSV → preprocess.py cleans & scales it → train_model.py trains RandomForest on 80% of data → saves model.pkl, scaler.pkl, label_encoder.pkl
2
Stream (runs continuously)
kafka_producer.py generates a simulated flow every ~1 second with 20 feature values → producer.send("network-flows", flow) → Kafka stores the message in the topic queue
3
Infer (runs continuously — future implementation)
backend/app.py Kafka consumer reads next flow → scaler.transform([flow.features]) → model.predict() → label_encoder.inverse_transform() → builds alert dict → socketio.emit('new_alert', alert)
4
Display (runs in browser)
React App.js receives new_alert event → setAlerts([alert, ...prev]) → React re-renders Dashboard → table row appears, chart animates, counter increments — all in <50ms
🔧
Current state (simulation)

Steps 2 & 3 are currently simulated — backend/app.py uses random.choices() and random.uniform() instead of Kafka + real ML inference. This lets the dashboard work end-to-end while the ML pipeline is still being built. The architecture is identical — only the data source changes when you wire it up for real.

Key Terms Glossary
Quick reference for every concept in this project
IDS
Intrusion Detection System — monitors network for malicious activity
Network flow
Statistical summary of a connection (not raw packets)
CIC-IDS2017
Standard academic dataset of 2.8M labelled network flows
Random Forest
Ensemble of 100 decision trees that vote on a classification
LabelEncoder
Converts text class names → integers for ML
StandardScaler
Normalises features to mean=0, std=1

Using test/future data during training — makes accuracy fake
Overfitting
Model memorises training data but fails on new data
Pickle
Python's way of saving/loading any object to/from disk
Stratify
Keep class ratios identical in train and test splits
Precision
Of predicted attacks, fraction that were truly attacks
Recall
Of real attacks, fraction that were correctly caught
F1-Score
Harmonic mean of precision and recall
Kafka
Distributed message broker for high-speed event streaming
Topic
Named channel in Kafka that producers write to and consumers read from
Offset
Kafka remembers where a consumer left off — no data lost on restart
Flask
Lightweight Python web framework for building APIs
CORS
Browser security policy — Flask-CORS allows cross-origin requests
WebSocket
Persistent two-way connection — server can push data at any time
Socket.IO
Wraps WebSocket with auto-reconnect and named events
Daemon thread
Background thread that dies when the main program exits
React
JS library for building component-based UIs
useState

useEffect

Docker

Volume mount
